## Critical Findings

### P0 - Missing GSD milestone evidence prevents standard requirement coverage audit

The GSD initializer returned `roadmap_exists=false`, `phases_dir_exists=false`, `phase_count=0`, and `completed_phases=0`. `node ... gsd-tools.cjs phases list` returned an empty phase list.

Because there are no milestone requirements, phase summaries, or phase verification files, the required 3-source matrix cannot be computed:

- REQUIREMENTS traceability table
- phase `VERIFICATION.md` requirement tables
- phase `SUMMARY.md` `requirements-completed` frontmatter

This is a process blocker, not a business-code bug. It means milestone archive/closure cannot be evidenced through the GSD contract.

### P0 - Backend release gate is red

The canonical release suite is defined in `scripts/backend_release_suite.py` and is the backend CI gate in `.github/workflows/ci.yml`.

Observed failure from targeted release subset:

```text
tests/test_balance_analysis_api.py::test_balance_analysis_current_user_api_uses_same_auth_context_as_status_write
expected role: reviewer
actual role: viewer
```

Relevant code evidence:

- `backend/app/api/routes/balance_analysis.py:223` exposes `/ui/balance-analysis/current-user`.
- `backend/app/api/routes/balance_analysis.py:225` depends on `get_auth_context`.
- `backend/app/security/auth_context.py:24` accepts `X-User-Id` from request headers.
- `backend/app/security/auth_context.py:37` only trusts `X-User-Role` when `MOSS_AUTH_TRUST_X_USER_ROLE_FOR_DEV_TEST` is enabled.
- `tests/test_balance_analysis_api.py:457` expects the current-user endpoint to use the same role context as status write.

Impact: the governed backend release suite is not green, and the failing area is an auth-context surface used by business workflow UI.

Impact: these routes are visible/live but cannot be traced through the page-level contract evidence model.

### P0 - Frontend finance-boundary guard is red

The guard in `tests/test_no_finance_logic_in_frontend.py` scans non-allowlisted frontend source for formal finance tokens such as `DV01`, `KRD`, `CS01`, `convexity`, `FVTPL`, and `FVOCI`.

Observed issue:

- `frontend/src/features/workbench/dashboard/dashboardCockpitModel.ts:482` contains `组合DV01`.
- `frontend/src/features/workbench/dashboard/dashboardCockpitModel.ts:740` contains `DV01`.
- `frontend/src/features/workbench/dashboard/dashboardCockpitModel.ts:843` contains `DV01`.

Impact: either the dashboard cockpit model now contains finance-domain display terms outside an allowed display-only boundary, or the guard whitelist needs a deliberate update. As written, the guard is failing.

### P1 - Header identity and authorization boundaries are not production-safe

Security review found several high-risk trust-boundary issues:

| Issue | Evidence | Risk |
|---|---|---|
| Client-controlled identity | `backend/app/security/auth_context.py:24` trusts `X-User-Id` | Caller can spoof user identity where routes rely on `auth.user_id` |
| Macro refresh lacks authorization check | `backend/app/api/routes/macro_vendor.py:72` accepts auth but does not call `ensure_user_allowed` | Any reachable caller can trigger refresh side effects |
| Agent run status lacks auth dependency | `backend/app/api/routes/agent.py:129` returns run status by run_id without ownership check | Prior prompts/results/tool outputs can leak if run_id is known |
| GitNexus repo path reaches local command config | `backend/app/services/gitnexus_service.py:175` accepts request repo paths; `backend/app/services/gitnexus_mcp_client.py:141` reads that repo's `.mcp.json`; line 51 launches the resolved command | If agent endpoints are enabled and attacker controls a repo path, local command execution boundary is weak |

Impact: treat the API as internal/dev only until a real authentication and trusted-repo boundary is enforced.

### P1 - Dashboard first-screen metric semantics can regress

Current uncommitted dashboard changes introduce shared formatting:

- `frontend/src/features/workbench/dashboard/dashboardHomeModel.ts:182` only falls back when delta display is blank.
- `frontend/src/features/workbench/dashboard/dashboardHomeModel.ts:203` uses this formatter for hero metric deltas.
- `frontend/src/features/workbench/pages/DashboardPage.tsx:467` passes the governed fallback label.

Risk from code review: existing dashboard/mock metric data uses the dash placeholder (`"—"`) for missing deltas. The old behavior converted missing/placeholder deltas to the read-chain/source label; the new formatter can surface `"—"` directly on the first screen.

Impact: the dashboard hero strip can lose provenance context exactly when data is partial, mock, or unavailable.

## Recommended Fix Order

1. Restore backend release suite green status by resolving the Balance Analysis auth-context role mismatch. Decide explicitly whether role headers are dev-test-only or whether `/current-user` should share the status-write test auth context.
2. Close page-contract gaps for `/agent`, `/balance-movement-analysis`, and `/liability-analytics`, or mark them as temporary exceptions if they are not ready for governed page contracts.
3. Resolve the frontend finance-boundary guard by either moving dashboard DV01 semantics into an allowed display-only contract boundary or updating the guard with a documented exception.
4. Fix the dashboard hero delta placeholder regression by treating dash placeholders as missing and adding a focused model test.
5. Harden high-risk auth/trust boundaries: do not trust arbitrary `X-User-Id` in production paths, gate macro refresh, add auth/ownership to agent run status, and restrict GitNexus repo path plus MCP command resolution.
6. Add `npm run build` to frontend CI and include important repo-level guards in CI so green CI better matches release confidence.
7. Initialize or restore `.planning` milestone artifacts if GSD milestone closure is expected: ROADMAP, REQUIREMENTS traceability, phase SUMMARY, phase VERIFICATION, and VALIDATION files.

## Remaining Risks

- No full browser E2E was run against a fresh backend/frontend pair during this audit.
- No full backend test suite was run in a single local command by the orchestrator; the test-engineer lane ran the canonical release suite and reported one failure.
- The worktree is dirty with many unrelated untracked screenshots and modified files. This audit intentionally did not revert or clean them.
- Some security findings may be acceptable for an explicitly local-only system, but they must be documented as local-only controls before promotion.

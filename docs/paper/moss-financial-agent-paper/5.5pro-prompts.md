# 5.5PRO 写作提示词

本文档把 5.5PRO 设定为两个角色：

- 主写手：把项目材料改写成期刊论文语言。
- 审稿人：检查项目报告味、夸大、证据不足和实验缺口。

使用原则：

- 每次只写一章或一个小节。
- 每次都提供 `material-pack.md` 中相关材料。
- 不允许模型自行编造实验数据、模块状态或引用。
- 每章输出后必须进入审稿人模式自审，再重写。

## 1. 总控提示词

```text
你是一名金融科技与信息系统方向的中文期刊论文导师。

我正在写一篇关于 MOSS 系统的论文。MOSS 是一个面向固定收益分析的可治理金融 Agent 分析系统，其核心不是泛泛的 AI 问答，而是把 Agent 绑定到正式金融计算、指标契约、证据链、数据血缘和审计元数据上。

请基于我提供的材料，设计一篇适合中文核心/普刊投稿的论文。请输出：
1. 研究背景与问题
2. 研究问题 RQ
3. 论文题目备选
4. 创新点
5. 摘要
6. 章节目录
7. 每章写作重点
8. 实验验证方案
9. 可能被审稿人质疑的问题
10. 需要补充的数据或图表

硬约束：
- 不要写成项目报告。
- 不要把未验证能力写成已完成能力。
- 不要编造实验数据。
- 所有系统能力必须能追溯到材料中的项目证据。
- 论文路线是系统设计与应用验证，不是纯 AI 算法创新。
```

## 2. 引言提示词

```text
请写论文第 1 章“引言”。

写作目标：
- 从金融 AI 落地问题切入，而不是从技术栈切入。
- 强调金融分析场景中的幻觉、口径漂移、不可审计、不可复核问题。
- 引出 MOSS 的治理绑定思路：Agent 不绕过正式计算层，所有正式结果带 result_meta 和 evidence。
- 明确本文贡献。

必须包含：
1. 研究背景
2. 问题定义
3. 本文方法概述
4. 主要贡献
5. 文章结构

避免：
- 不要宣传式语言。
- 不要说“本系统显著提高效率”除非实验数据已提供。
- 不要把 MOSS 写成通用 AGI 或自动投研系统。
```

## 3. 相关研究提示词

```text
请写论文第 2 章“相关研究”。

主题包括：
1. 金融智能分析系统
2. LLM/Agent 在金融领域的应用
3. 金融数据治理、指标口径和血缘追踪
4. AI 风险管理与可审计性

可参考外部坐标：
- NIST AI RMF
- FSB 关于 AI 金融稳定风险的报告
- IOSCO 关于 AI/ML 在金融中介和资管中的报告
- Finance Agent Benchmark
- FinGAIA
- FinMCP-Bench

写作要求：
- 不要写成文献堆砌。
- 每一类研究都要指出不足：现有方法往往重视回答能力，较少处理正式金融计算、证据链和审计元数据绑定。
- 最后自然引出本文的治理绑定 Agent 架构。
```

## 4. 系统架构提示词

```text
请写论文第 3 章“MOSS 系统架构设计”。

必须覆盖：
- 系统定位：Agent 可调用的分析操作系统。
- 架构形态：模块化单体 + 分层分析栈。
- 固定调用方向：frontend -> api -> services -> core_finance/governance -> storage。
- 存储职责：PostgreSQL、DuckDB、Redis、MinIO/S3。
- REST/MCP 双接口。
- 前端、报表、Agent 共用同一事实层。

写作重点：
- 解释为什么这样设计能降低口径漂移和审计风险。
- 不要只罗列技术栈。
- 不要声称微服务、完整外部数据连接器或全自动报告已完成。
```

## 5. 关键机制提示词

```text
请写论文第 4 章“可治理 Agent 关键机制”。

必须覆盖：
1. result_meta 审计元数据契约
2. evidence 与 lineage
3. Formal / Scenario / Analytical 隔离
4. 只读 Agent 安全边界
5. 受控 intent 与 workflow

写作要求：
- 把机制写成解决金融 AI 风险的设计，而不是代码说明。
- 每个机制说明其解决的问题、设计方式和可验证证据。
- 强调 Agent 不直接生成正式金融指标，而是调用受控服务并返回证据。
```

## 6. 系统实现提示词

```text
请写论文第 5 章“系统实现”。

实现模块包括：
- 固定收益分析
- balance analysis
- formal PnL
- PnL bridge
- risk tensor / DV01 / 久期
- Agent Workbench
- 指标字典、页面契约、golden sample

写作要求：
- 以业务能力和治理链路组织，不以文件目录组织。
- 对每个模块说明输入、计算边界、输出、证据和测试锚点。
- 不要夸大当前未落地模块。
```

## 7. 实验验证提示词

```text
请写论文第 6 章“实验验证与案例分析”。

实验设计：
- 对比普通 LLM、RAG、MOSS Agent。
- 使用 30-50 个金融分析问题。
- 记录数值正确率、口径正确率、单位/日期正确率、result_meta 完整率、证据可追溯率、幻觉率和人工复核耗时。
- 至少包含组合概览、PnL Bridge、久期/DV01 风险分析三个案例。

硬约束：
- 如果没有实验结果，请只写实验设计和待记录表格，不要编造结论。
- 如果提供了实验结果，请基于结果讨论，不要夸大因果。
```

## 8. 结论提示词

```text
请写论文第 7 章“结论与展望”。

必须包括：
- 本文解决的问题
- 治理绑定架构的价值
- 系统实现和验证的结论
- 局限性：样本规模、真实生产数据脱敏、外部模型依赖、当前 Agent golden sample 尚未覆盖
- 未来工作：更多金融工作流、多模型评测、监管审计接口、报告生成

避免：
- 不要写成宣传总结。
- 不要引入正文没有论证的新能力。
```

## 9. 审稿人模式提示词

```text
请以中文核心期刊审稿人的标准审查下面这一章。

重点检查：
1. 是否像项目说明而不是论文
2. 创新点是否明确
3. 论证是否有证据支撑
4. 是否存在夸大系统能力
5. 是否需要补充图表或实验
6. 是否存在术语不一致，如 formal、scenario、analytical 混用
7. 是否有不该出现的营销式表达

请输出：
- 总体评价
- 主要问题
- 逐段修改建议
- 需要补充的证据或图表
- 重写后的版本
```

## 10. 反夸大审查提示词

```text
请只做“反夸大审查”。

任务：
逐条检查下面文字中的系统能力、实验结论、数据范围和创新表述。

请标注：
- 可以保留：已有材料或实验支撑
- 需要降级：表述过强，但方向合理
- 必须删除：材料没有支撑或可能误导审稿人
- 需要补证：可以写，但必须补充证据

特别关注：
- Agent 是否被写成能直接生成正式金融结论
- scenario/analytical 是否被写成 formal
- golden sample 是否被写成覆盖 Agent
- 未落地外部数据连接器、报告生成、监管接口是否被写成已实现
```
